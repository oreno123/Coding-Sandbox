## 1 项目背景与目标
### 1.1 背景与问题
南京大学图书馆藏书丰富，馆区众多。要想让师生充分利用里面的资源，一个易用的图书检索平台尤为重要。官方微信公众号中提供了图书馆服务平台。在实际使用过程中，可以发现该平台在信息展示和交互体验等方面仍存在一定改进空间。例如：

+ 无法按照校区筛选结果。要想知道自己所在的校区有没有相关藏书，用户需要逐一点击搜索结果，并自行确认；
+ 无法查找特定作者或出版社的图书；
+ 检索列表中的多条结果可能对应同一种图书；
+ 当图书包括多个版本/分册/分期以及较多实体册数时，馆藏信息显示的顺序感不明显，不便查找。

### 1.2 目标
鉴于上述问题，本项目希望对检索平台做以下优化：

+ 设计更优雅的前端；
+ 增加按照校区、作者和出版社搜索功能；
+ 合并实际相同的图书和相同地点的馆藏记录，并按顺序呈现。

## 4 总结
### 4.1 不足与优化方向
本项目仍存在诸多不足，可供优化。例如：

+ 受streamlit原理及官方服务器所限，该版本存在诸多不便：
  - 搜索结果数量过多时，页面加载速度慢；
  - 任何输入改变都会触发页面重置；
  - 一段时间内无人访问，应用会休眠。
+ 缺少封面图片这一直观元素。
    - 对原网页的请求中必须包含 `openid` 参数才能够加载出封面图片，从而进一步抓取。而 `openid`与个人微信账号和读者证号有关，相关行为不明确，推测用于鉴权，具有隐私性。鉴于此，本项目不尝试通过此途径抓取封面图片；
    - 项目开发过程中曾尝试使用免费API，通过书号请求得到封面。但调试过程中发现该API缺失大量国内图书封面数据，故弃用；
    - 优化时可尝试使用质量更高的相关API，或进一步研究原网页加载逻辑，尝试获取封面图片。
+ 相似馆藏信息合并
    - 在部分搜索结果中，多条藏书信息（`Copy`）仅有编号不同，索书号、状态信息与版本/分册/分期信息等都相同。本项目中，这样的一些藏书信息依然会逐条展示；
    - 考虑实际应用情景。编号的区别对用户查找图书基本没有影响，可以将具有上述特征的藏书信息的编号部分合并，展开可一一查看。
+ 搜索记录
    - 用户无法在程序中保存自己需要的馆藏记录备用；
    - 可加入账号注册功能或使用 cookie 等，保留用户的搜索记录。
+ 字典序
    - 对藏书信息（`Copy`）的排序是根据各属性的字典序的，但与实际分期顺序等并不完全一致；
    - 可接入大模型进行智能排序。
+ 状态图标与颜色
    - 现有的状态图标与颜色，是根据实际可能出现的状态文本一一设置的，覆盖了绝大多数情况，但假如抓取到预设之外的状态文本，则会显示为普通文本；
    - 可接入大模型智能设计新状态的图标与颜色。
